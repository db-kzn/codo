# Руководство по развертыванию Blazor WebAssembly приложения

## Введение

Это руководство описывает различные способы развертывания Blazor WebAssembly приложения на различных платформах. Проект уже настроен для развертывания на GitHub Pages с учетом всех требований этой платформы.

## Поддерживаемые платформы

1. GitHub Pages (основная платформа)
2. Статические хостинги (Netlify, Vercel, и т.д.)
3. Любой веб-сервер, способный обслуживать статические файлы

## Развертывание на GitHub Pages

### Подготовка к развертыванию

Проект уже содержит все необходимые файлы и настройки для развертывания на GitHub Pages:

1. `404.html` - файл для корректной работы маршрутизации SPA
2. `.nojekyll` - файл для отключения обработки Jekyll
3. JavaScript код в `App.razor` для поддержки маршрутизации SPA
4. Настроенный prerendering через пакет BlazorWasmPreRendering.Build

### Автоматическое развертывание через GitHub Actions

Проект включает готовый workflow для автоматического развертывания на GitHub Pages:

1. Создайте репозиторий на GitHub и отправьте в него код проекта
2. Перейдите в настройки репозитория (Settings → Pages)
3. В разделе "Source" выберите "GitHub Actions"
4. При следующем пуше в ветку `main` автоматически запустится workflow

Файл workflow находится в `.github/workflows/deploy.yml` и включает:
- Сборку проекта с помощью .NET 9.0
- Подготовку файлов для развертывания
- Автоматическую публикацию на GitHub Pages

### Ручное развертывание

Если вы предпочитаете ручное развертывание, вы можете использовать один из следующих способов:

#### С использованием скриптов

1. Запустите соответствующий скрипт для вашей операционной системы:
   - Linux/macOS: `./build-and-deploy.sh`
   - Windows: `build-and-deploy.bat`

2. Скрипты создадут директорию `temp_deploy` с подготовленными файлами

3. Отправьте содержимое этой директории в ваш репозиторий GitHub

#### Без использования скриптов

1. Выполните сборку проекта:
   ```bash
   cd Host
   dotnet publish -c Release
   ```

2. Скопируйте содержимое директории `Host/bin/Release/net9.0/publish/wwwroot` в отдельную директорию для развертывания

3. Убедитесь, что в директории для развертывания присутствуют следующие файлы:
   - Все файлы из сборки
   - `404.html`
   - `.nojekyll`

4. Отправьте файлы в ваш репозиторий GitHub

### Настройка GitHub Pages

1. Перейдите в настройки репозитория на GitHub (Settings → Pages)
2. В разделе "Source" выберите ветку для развертывания (обычно `main` или `gh-pages`)
3. Сохраните настройки и дождитесь завершения развертывания

## Развертывание на других платформах

### Netlify

1. Создайте аккаунт на Netlify
2. Подключите ваш репозиторий GitHub
3. Настройте параметры сборки:
   - Build command: `cd Host && dotnet publish -c Release`
   - Publish directory: `Host/bin/Release/net9.0/publish/wwwroot`
4. Запустите сборку

### Vercel

1. Создайте аккаунт на Vercel
2. Подключите ваш репозиторий GitHub
3. Настройте параметры сборки:
   - Build Command: `cd Host && dotnet publish -c Release`
   - Output Directory: `Host/bin/Release/net9.0/publish/wwwroot`
4. Запустите сборку

### Статический хостинг

Для развертывания на любом статическом хостинге:

1. Выполните сборку проекта:
   ```bash
   cd Host
   dotnet publish -c Release
   ```

2. Скопируйте содержимое директории `Host/bin/Release/net9.0/publish/wwwroot` на ваш хостинг

3. Убедитесь, что ваш веб-сервер настроен на обслуживание статических файлов

## Особенности развертывания SPA

При развертывании Single Page Application необходимо учитывать следующие особенности:

1. Все неизвестные маршруты должны возвращать `index.html`
2. Для GitHub Pages это реализовано через файл `404.html`
3. Для других платформ может потребоваться специальная настройка веб-сервера

## Устранение неполадок

### Приложение не загружается

1. Проверьте наличие файла `.nojekyll` в корне развертывания
2. Убедитесь, что файл `404.html` присутствует
3. Проверьте консоль браузера на наличие ошибок загрузки ресурсов

### Ошибки маршрутизации

1. Убедитесь, что JavaScript код для маршрутизации присутствует в `App.razor`
2. Проверьте, что файл `404.html` корректно настроен для перенаправления

### Проблемы с базовым URL

Если приложение развернуто в подкаталоге (например, `username.github.io/repository`):

1. Может потребоваться настройка базового URL в файле `App.razor`
2. Убедитесь, что все пути в приложении относительные
